# Лабораторная работа: Сравнительный анализ производительности REST API фреймворков

### Цель работы

Закрепить навыки нагрузочного тестирования веб-приложений, научиться самостоятельно сравнивать производительность различных фреймворков (синхронных и асинхронных), анализировать влияние блокирующих операций на пропускную способность, формулировать рекомендации по оптимизации, а также освоить применение LLM-ассистента **GigaCode** для генерации кода и тестовых сценариев с использованием техник zero-shot, one-shot и few-shot.

### Задачи

1. Реализовать тестовое REST API на **трёх** фреймворках по выбору (минимум один синхронный и один асинхронный, кроме FastAPI).
2. Для каждого фреймворка создать два эндпоинта:
   - `/cpu` – выполняет CPU-интенсивную задачу (суммирование 10–20 млн чисел) **синхронно** (блокирующий режим).
   - `/cpu_fixed` – выполняет ту же задачу, но **неблокирующим** способом (вынос в поток/процесс для асинхронных фреймворков; для синхронных – использовать асинхронные библиотеки или показать невозможность).
3. Провести нагрузочное тестирование с помощью **Locust** (или **k6**, **JMeter**) для каждого эндпоинта при 50, 100 и 200 одновременных пользователях.
4. Собрать метрики: RPS, средняя latency, p95, p99, процент ошибок.
5. Построить сводные таблицы и графики сравнения.
6. Сформулировать общие принципы стресс-тестирования на основе полученных результатов.
7. **Использовать GigaCode** для генерации части кода и сценариев, применяя zero-shot, one-shot и few-shot подходы. Задокументировать полученные промпты и результаты.

### Требования к реализации

- Все серверы должны запускаться на одной машине (или в контейнерах) с одинаковыми ресурсами (CPU, RAM).
- Количество воркеров/потоков должно быть одинаковым для честного сравнения (например, 1 воркер / 1 поток).
- Время выполнения CPU-задачи должно быть одинаковым во всех фреймворках (фиксированное число итераций).
- Тестирование проводить в **headless**-режиме Locust с фиксированным временем (например, 3 минуты на сценарий).

### Рекомендуемые фреймворки для сравнения

| Тип | Фреймворк | Комментарий |
|-----|-----------|-------------|
| Синхронные | Flask, Bottle, Django (WSGI) | Классические, простые в реализации |
| Асинхронные | Sanic, Tornado, Quart, aiohttp | Современные, основанные на asyncio |
| Другие языки (опционально) | Node.js (Express), Go (Gin), Java (Spring Boot) | Для расширенного сравнения |

> **Важно:** Если вы выбираете фреймворк на другом языке, вы должны сами реализовать нагрузочное тестирование (Locust поддерживает HTTP-запросы к любому хосту).

### Использование GigaCode при выполнении работы

В процессе разработки вам необходимо применить **GigaCode** (аналог GitHub Copilot) для автоматической генерации кода. Вы должны **сознательно** использовать три техники промптинга и задокументировать их:

| Техника | Описание | Пример применения |
|---------|----------|-------------------|
| **Zero-shot** | Запрос без примеров. Вы просто описываете задачу на естественном языке. | «Напиши на Python эндпоинт для Flask, который суммирует числа от 1 до 10 миллионов и возвращает результат в JSON.» |
| **One-shot** | В запросе приводится **один** похожий пример кода, на основе которого GigaCode генерирует решение. | «Вот пример эндпоинта на FastAPI: … Теперь сделай такой же на Sanic.» |
| **Few-shot** | В запросе приводится **2-3** разнородных примера, чтобы задать стиль, паттерны или структуру. | «Пример 1: … Пример 2: … Сгенерируй эндпоинт для Tornado с аналогичной логикой.» |

**Что нужно сделать:**

1. **Сгенерировать код приложений** для двух из трёх выбранных фреймворков с помощью GigaCode, используя разные техники:
   - Для одного фреймворка – **zero-shot**.
   - Для второго – **one-shot** (предварительно подготовив пример на другом языке или фреймворке).
   - Для третьего – **few-shot** (например, показав два примера обработки CPU-задач с выносом в поток).

2. **Сгенерировать сценарий нагрузочного тестирования (locustfile.py)** с помощью GigaCode. Попробуйте разные техники, чтобы получить параметризованный скрипт, поддерживающий переменное количество пользователей и время теста.

3. **Сгенерировать код для построения графиков** (например, на matplotlib) на основе CSV-отчётов Locust. Сравните качество результатов при zero-shot и few-shot.

4. **Задокументировать все использованные промпты** в отчёте:
   - Что вы запросили у GigaCode.
   - Какую технику применили.
   - Что получили в ответе (полный сгенерированный код или фрагменты).
   - Пришлось ли вручную дорабатывать код? Почему?

5. **Проанализировать эффективность** каждой техники:
   - Какая техника дала наиболее рабочий код с первого раза?
   - Для каких задач (создание эндпоинтов, тестов, графиков) лучше подходит zero-shot, а для каких – few-shot?
   - Были ли ситуации, когда GigaCode генерировал неверный или неоптимальный код (например, блокирующий вызов в асинхронном фреймворке)?

**Ожидаемый результат:**  
В отчёте должен присутствовать отдельный раздел **«Применение GigaCode»** с таблицей промптов, полученными фрагментами кода и выводами о применимости техник zero-shot, one-shot, few-shot для задач веб-разработки и нагрузочного тестирования.

### Реализация REST API


**Flask**

**Эндпоинт /cpu (блокирующий режим):**

```
@app.route('/cpu', methods=['GET'])
def cpu_bound():
    iterations = int(request.args.get('n', 15_000_000))
    start = time.time()
    result = cpu_intensive_task(iterations)
    elapsed = time.time() - start
    return jsonify({'result': result, 'time_seconds': elapsed})
```

**Эндпоинт /cpu_fixed:**

Для синхронного фреймворка не существует неблокирующего режима для CPU-задач. Эндпоинт демонстрирует идентичное поведение с пояснением в ответе.

**Tornado (асинхронный фреймворк)**

Эндпоинт /cpu (блокирующий режим — антипаттерн):

```
class CPUBoundHandler(tornado.web.RequestHandler):
    def get(self):
        iterations = int(self.get_argument('n', 15_000_000))
        result = cpu_intensive_task(iterations)  # Блокирует IOLoop!
        return json.dumps({'result': result})
```
**Эндпоинт /cpu_fixed (неблокирующий режим — правильный):**

```
class CPUFixedHandler(tornado.web.RequestHandler):
    async def get(self):
        loop = tornado.ioloop.IOLoop.current()
        result = await loop.run_in_executor(executor, cpu_intensive_task, 15_000_000)
        return json.dumps({'result': result})
```


**aiohttp (асинхронный фреймворк)**

Эндпоинт /cpu (блокирующий режим):

```
async def cpu_bound(request):
    iterations = int(request.query.get('n', 15_000_000))
    result = cpu_intensive_task(iterations)  # Блокирует event loop!
    return web.json_response({'result': result})
```

**Эндпоинт /cpu_fixed (неблокирующий режим):**

```
async def cpu_fixed(request):
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(executor, cpu_intensive_task, 15_000_000)
    return web.json_response({'result': result})
```

### Применение GigaCode (LLM-ассистент)

| Техника | Для чего | Промпт | Полученный код | Доработки |
|---------|----------|-------------------|-------------|--------|
| Zero-shot | Flask-эндпоинт | Напиши на Python эндпоинт для Flask, который суммирует числа от 1 до 10 миллионов синхронно и возвращает результат в JSON | Базовый код с эндпоинтом /cpu и циклом for | Добавил query-параметр n для возможности изменения количества итераций, добавил измерение времени выполнения |
| One-shot | Tornado | Вот пример на FastAPI: async def compute(): loop.run_in_executor(executor, sync_func). Теперь сделай такой же на Tornado | Корректный код с run_in_executor | Исправил способ получения event loop (tornado.ioloop.IOLoop.current() вместо asyncio.get_running_loop()), добавил executor с фиксированным размером пула |
| Few-shot | aiohttp + locustfile | код на Tornado с run_in_executor. Создай аналогичный на aiohttp. Также создай locustfile с двумя задачами для /cpu и /cpu_fixed | Полноценный код для aiohttp и параметризованный locustfile | Добавил health check в locustfile, настроил логирование, исправил импорты |


### Проведение нагрузочного тестирования

Тестирование проводилось с помощью Locust в headless-режиме:

```
locust -f locustfile.py --headless -u 50 -r 10 --run-time 3m --host http://localhost:5000 --csv=reports/flask_50_cpu
```

Аналогично для tornado и aiohttp для 50, 100 и 200 пользователей

**Сложности при тестировании**

1. Проблема с aiohttp: Неблокирующий режим работал некорректно даже после выгрузки в ThreadPoolExecutor — наблюдалось 80-95% ошибок при 100-200 пользователях
2. Ошибка ConnectionResetError в aiohttp: Клиенты разрывали соединение до получения ответа из-за слишком долгого ожидания

### Ключевые выводы

* Синхронный Flask — стабильно низкая производительность (~1 RPS). При росте нагрузки latency растёт линейно, health check начинает падать.

* Tornado (неблокирующий) показывает наилучшие результаты:

- RPS растёт с нагрузкой (560 → 1080 → 2050)

- Latency остаётся стабильно низкой (~90-100 мс)

- Нулевой процент ошибок

* Блокирующий режим в асинхронном фреймворке (Tornado /cpu) — хуже синхронного! При 200 пользователях RPS падает до 48, ошибки достигают 22.7%.

* aiohttp показал неожиданно плохие результаты — даже неблокирующий режим работал некорректно на Windows 11.

* Лучший выбор для CPU-интенсивного API: Tornado с неблокирующим режимом

* Стабильный выбор для простых API: Flask
